Method documentation

Destroy

Destroy, masking or wiping out part of the data, is the basic way of anonymizing data. Columns, that are not relevant for your anonymized data set, as the street name in a client record, can be wiped out, or - preferably - replaced by some placeholder dummies.

Please note, that masking is not a constraint tolerant annotation. Unique and referential column constraints will be dropped to run this type of anonymisation.

Encoding

Encoding a column with a seed (password) allows for tracing back the data for the password owners. This compromise of anonymity is sometimes desirable, as data errors or method errors may need to be traced back to the original de-anonymized data, to allow for error correction and data discrepancies elimination.

Encoding is a better option with regard to constraints. Uniqueness of the values can be maintained, and referential integrity will be preserved, if the referenced table is also included in the anonymisation.

Reshuffle

Reshuffle is a method of re-shuffling your data, that allows for very good human readability that is often required in testing or acceptance scenarios. Original values of the to-be-anonymized entities are replaced by a pseudo-random replacement value of all possible values in the domain. A seed (password) is used again to re-shuffle the values; hence de-anonymisation for error analysis is possible for password owners.

If the database defines referential or unique constraints for the target columns, reshuffle is the best option. These constraints can be reactivated successfully in nearly all scenarios.

Map

Mapping based anonymisation can be used in scenarios, where a larger number of values must be compressed to a smaller set of values. A good example would be the age of an employee. The exact age may be rounded to the next multiple of 10.

The home address of a customer may be mapped to a space of 5-10 hypothetical addresses.

While referential constraints can be maintained by this method, the existing unique constraints will nearly always be dropped.

Architecture Picture

Configuring

Creating a database configuration

The first step in creating an anonymisation configuration is defining the connection to the target database. The database type, the JDBC connection URL and the user/password must be provided. Anonimecs will validate the connection, and let you define your target schema in this database.

The next step in the configuration workflow is connecting the database. After connecting the database, the metadata dictionaries will be loaded, and the user is presented a Database Browser like view of the target schema. Selecting alternative schemas is also supported.

The user will also be presented a list of suspicions columns, where the name of the column and its data type suggest sensitive information.

The user will now select the tables that need to be anonymized on by one, and conclude the following steps.

Anonymisation method

After selecting the target column in the target table, the user will select an anonymisation type. Refer to Supported databases, methods and data types chapter for details on supported methods. One of the key features of Anonimecs is to always visualize your actions. When selecting the anonymisation method, the GUI will display sample 5 rows of the column values, in the original form, and anonymized. This way the user will always have good understanding what to expect from this anonymisation.

Some anonymisations, like mapping require further fine tuning that is achieved in a popup window.

Anonymized column and related tables

One of the main benefits of Anonimecs over the custom made SQLs is the support of database constraint. The database may constrain the values of the target column in many ways, like referential integrity (incoming or outbound), primary key, unique, etc. Anonimecs has sophisticated metadata handling to cater for these constraints. One example of this is the use of foreign keys as a mean to gather related tables into the same anonymisation method. If the user selects columnA in tableA, that is referentially linked to columnB in tableB, then Anonimecs will automatically suggest including columnB in the same anonymisation method as columnA.

This ensures, that referential integrity is maintained after the anonymisation run. Alternatively, the user may decide to exclude a referentially linked table from the anonymisation method. In this case, Anonimecs will need to drop or deactivate the referential constraint during the anonymisation run.

Anonymisation run

Once you have finished setting up your configuration, or when you want to check the validity of your anonymisation settings, you will need to run your configuration. You will do this in the Run->Execute menu.

As described in detail in the previous chapter, in Anonimecs, the basic unit of execution is one anonymisation method comprising one or more anonymized columns. In enterprise version you can selectively run these methods, by clicking on the run button of the method. The changes in the anonymisation process are pushed to the execution GUI immediately, allowing for following the process of anonymisation without database browser tools.

The anonymisation SQL, generated by Anonimecs on the fly, is one of the main features of the application. Most of the anonymisation methods are based on SQL functions and procedures, that are generated, pushed on to the target database, executed, and cleaned up. This approach allows for enhanced performance optimization, such as usage of indices, temporary suspension of constraints, to yield a fast anonymisation run.

During the anonymisation run, Anonimecs will try to keep as many constraints as possible. The user may have however chosen to violate some referential or unique constraints with the configuration. Such constraints will be suspended ( for example in Oracle) or removed (for example in Sybase), depending on the vendor specific possibilities. See the next chapter on more details on constraint management.

Running all anonymisation or scheduling the anonymisation via the Scheduler will execute the whole batch of anonymisation methods defined in the connected database configuration.

Each anonymisation run will leave a detailed audit trail as described in the Auditing section.

Constraint Management

Constraints are the essence of relational databases, and database developers go to great lengths to ensure the business data is constrained to the business domain. The database vendors that are supported by Anonimecs will each implement the commonly used data constraints, while some vendors add vendor specific constraints (MYSQL Enum, Oracle check constraint). Anonimecs supports the following list of constraints, which are currently supported by all vendors:

  • Not Null Columns
  • Primary Keys (single and multi column)
  • Unique Constraints (single and multi column)

· Incoming Foreign key constraints (single and multi column)

· Outgoing Foreign key constraints (single and multi column)

In addition to the above, Anonimecs supports cross schema constraints for Oracle and MySql databases.

Constraint parsing and management is present in various parts of the Anonimecs process. The Config GUI will show the constraints for the selected column, aiding the configuration user to select the fitting anonymisation method. The Config GUI will also use the Foreign Key constraints to suggest binding referenced tables into a common anonymisation method.

During the anonymisation run, the constraint manager will suspend the constraints of the table. This has a significant speed impact, as the database will not need to validate the constraint for every changed row. In addition, some constraint may not be compatible with the targeted anonymisation. Such constraints will not be recreated. An example is the the user_mail column, that has a unique constraint, and the anonymisation configuration requires Destroy anonymisation no this column. Scheduling – Enterprise Feature

In Anonimecs, it is possible to run an anonymisation process in scheduled manner, without user interaction. The startServerRunner and the startStandaloneRunner scripts are provided with the enterprise distribution, with bat/sh content to cater for Windows/Linux.

The startStandaloneRunner.sh will not require a running Anonimecs server, and is intended for small to medium enterprises, where a dedicated server running the Anonimecs server may not be available.

Run example: startStandaloneRunner.sh salesdatabase

where salesdatabase is the GUI name of the database configuration you intend to anonymize.

The startServerRunner.sh script will connect to the server via the RMI API (see API section), and run the anonymisation within the context of the server. This is more feasible for large organization, with multiple database configurations.

Run example: startServerRunner.sh salesdatabase

where salesdatabase is the GUI name of the database configuration you intend to anonymize.

Enterprise tools for scheduling processes are well established by most of our client. We offer tested solutions to the most popular scheduling frameworks, including Cron, Quartz, UC4.

Multi user capabilities – Enterprise feature

While the free version of Anonimecs does not support securing the tool itself, the enterprise version comes with an extensible login feature. Out of the box, Anonimecs will install a default admin user admin/123456 that you can use for initial login to the GUI.

After the initial login, the admin should change the password of this admin user, and create other users representing the database configurators. Each database connection (configuration) will than be assigned to the user who created it. Please note, that users can share database configurations. Refer to the last chapter on how to achieve this.

Auditing

Logging and auditing is a common and proven concept of enterprise processes. It helps tracking the changes of your data, following the state of the process and looking for errors if your process does not work as intended.

Anonimecs offers powerful auditing mechanisms that track the state of your databases, data, and their intended anonymity. Each execution of an anonymisation method will leave a persistent trace of you action, be it a manual or scheduled execution. You can then review the data in the Run->Audit Logs menu point.

Any execution trigger will create a line in the displayed table. Clicking on this line will reveal the details of this anonymisation execution, and will help you to assess, where your process spent most of its time (database resources), and if it finished successfully. In case of data error, or other malfunction of your configuration, you can trace here down to the columns that have succeeded in anonymisation, and the first column, where the execution halted. Anonimecs will always halt the anonymisation process on the first error, as it is easier to fix this error and continue from this point, than deal with multiple errors and partial failures.

Auditing also helps to estimate runtimes of your anonymisation. Previous runtimes will be consulted when a new run is started, so you will have a relatively precise estimate of the expected runtimes.

Backup – Enterprise Feature

Sophisticated enterprises, with multiple databases, department and data streams require sophisticated anonymisation setups. These will span over several database configurations, may be provisioned by different users, and require days or weeks of implementation and testing. The Backup functionality comes to help here. It allows for regular dumps (backups) of the configuration data and retaining or moving this configuration.

You can think of this as a support for a change cycle in your anonymisation setup. Back up your current setup, as you are sure it works as intended. Make your configuration changes, until you are satisfied with the new configuration. Should you get lost on the way, just roll back to your last best setup, and start anew.

To create a backup of your current setup, navigate to the Admin->Backup menu. The displayed page will show you all the previous backups you or other admins have created. Press the Backup the configuration now button to create a new backup. This will back up the complete state of you Anonimecs, including anonymisation methods, users, audit trails.

You will not need to stop Anonimecs or the database server for this operation.

To roll back to a previous backup, go to the Admin->Backup menu, and follow the instructions in the bottom of the screen.

JAVA API – Enterprise Feature

The concepts of Anonimecs strongly discourages the one size fits all principle. We believe in scalable architecture, and offer solutions, that suit your business case complexity. One of the ultimate features of Anonimecs is its Java API that allows interacting with your installation solely with machine to machine manner.

The scheduling feature, described in the previous chapters makes itself use of this API. However, your scenarios are not limited to running an existing anonymisation configuration. You can create and manage your database connections or anonymisation configuration, and run them all as a part of your corporate application.

The Java API is centered on the anon-api.jar and an RMI interface to the Anonimecs Server Process. The following tutorial show you how to create a new anonymisation configuration with the use of Maven, Spring, Java and the anon-api.jar. This tutorial assumes you are already familiar with these Maven and Java.

1) Add maven dependency to the anon-api.jar available in your Anonimecs enterprise ZIP distribution. 

	<dependency>
		<groupId>org.anon</groupId>
		<artifactId>anon-api</artifactId>
		<version>${project.version}</version>
	</dependency>

This will transitively fetch all necessary spring dependencies.

2) Import the Spring RMI client configuration file spring-anon-rmi-client.xml your master spring config. 

    <import resource="classpath:spring-anon-rmi-client.xml"/>

This will load the RMI Stubs in the Spring container, and define the server lookup.

3) Inject your API stubs in your client application. 

    @Autowired DatabaseConfigService databaseConfigService;
    @Autowired EditedTableService editedTableService;

4) Start the Anonimecs server, and run your client code. 

	@Autowired	
	DatabaseConfigService databaseConfigService;

	@Autowired
	EditedTableService editedTableService;
	
	public void createDatabaseConfigration() {
		DatabaseConfig databaseConfig = createDatabaseConfigMySql();
		databaseConfigService.addDatabaseConfig(databaseConfig);
	}

	public DatabaseConfig createDatabaseConfigMySql() {
		DatabaseConfig config = new DatabaseConfig();
		config.setUrl("localhost:3306");
		config.setLogin("user");
		config.setPassword("password");
		config.setVendor(Database.MYSQL);
		config.setVersion(null);
		config.setGuiName("mysql-database");
		return config;
	}

	public void addAnonymisationMethod(){
		DatabaseTableInfo editedTable= new DatabaseTableInfo();
		editedTable.setName("dept_manager");
		editedTable.setSchema("employees");
		DatabaseColumnInfo editedColumn = new DatabaseColumnInfo("to_date", 
			"date", DatabaseSpecifics.MySqlSpecific);
		editedTable.addColumn(editedColumn);

		AnonymisedColumnInfo anonymizedColumn = new AnonymisedColumnInfo(editedColumn);
		
		AnonymisationMethod anonymisationMethod = new AnonymisationMethodDestoryMySql();
		
		
		DatabaseConfig databaseConfig = databaseConfigService.loadConnectionConfig(“mysql-database”);
		
		anonServer.useDatabaseConfig(databaseConfig);
		editedTableService.addAnonymisation(editedTable,
		anonymizedColumn,Collections.EMPTY_LIST, anonymisationMethod);
	}

5) This is all. After this point, your new database configuration is available in the Anonimecs GUI for manual configuration. Obviously, you can also configure the anonymisation over the API. For the full Java API documentation refer to the Javadoc of the anon-api.jar.


Configuration Database Access and SQL Batch Config – Enterprise Feature

Anonimecs stores its internal configurations in an SQL database that is also accessible via JDBC externally in the enterprise edition. You can use this access to export or import configurations, audits, or other table data. Worth noting is, that any sensitive information, like user passwords or database passwords are encrypted with the highest encryption in this database.

To access the config database use the following JDBC parameters in your database browser:

<property name="driverClassName" value="org.apache.derby.jdbc.ClientDriver" />
<property name="url" value="jdbc:derby://localhost:1527/anon" />
